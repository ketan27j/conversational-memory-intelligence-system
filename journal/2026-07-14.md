# 2026-07-14 — Project start: D1–D4 authored, Genesis spine set up, M0 built, M1 begun

## What happened

- Repository initialized with the full handbook structure (`reconstruction/`, `research/`,
  `design/`, `experiments/`, `implementation/`, `verification/`, `transfer/`, `journal/`,
  `.genesis/`) and Deliverables 1–4 (problem reconstruction, week-1 research scan, the naive
  baseline, and the system design docs including all 7 ADRs and the threat model) were already in
  place before the Genesis loop-based build started. `git log`'s first commit bundles all of this
  together as "Initial commit: D1–D4 deliverables + .genesis loop-ready spine."
- **Architecture decision (the biggest one of the project):** three approaches were on the table
  for the memory system —
  1. vector-only similarity search,
  2. hybrid search in one Postgres database (vector + keyword + entity) with an AI-judge write
     gate and recency/confidence-weighted contradiction resolution, and
  3. a bi-temporal knowledge graph with a separate vector/graph store and a pre-trained
     classifier write gate from day one.

  Approach 1 was rejected outright — Deliverable 3's own baseline measurement showed it misses
  exact facts (names, version numbers), the F4 failure. Approach 3 was rejected as overbuilt for
  a first version (no training data yet for a classifier without circular reasoning, a whole
  extra database to operate) and deferred behind a named trigger rather than abandoned — it's
  still the fallback if the recency/confidence rule ever fails a fact-change benchmark question.
  **Approach 2 (hybrid Postgres + AI-judge write gate) was chosen** and recorded as ADR-002/ADR-003.
- **M0 built and closed same day.** One Postgres database (isolated docker-compose stack, its own
  port, separate from an unrelated existing project's Postgres container on this host), row-level
  security on all 4 initial tables, skeleton `POST /messages`/`GET /memories` behind a
  verified-token auth stub.

## A real mistake, caught immediately

While wiring row-level security, `SET app.tenant_id = %s` was tried first — Postgres's `SET`
doesn't accept bind parameters at all, so this would have either failed outright or (worse) opened
an injection surface if someone had "fixed" it by string-formatting the tenant id in. The tests
caught it on the very first run; the fix was `SELECT set_config('app.tenant_id', %s, false)`,
which does accept a parameter. Small, but it's exactly the kind of mistake this journal is
supposed to surface rather than hide.

## L4 VERIFY (independent check)

A separate model instance, given only the goal and success criteria (not the build's own
reasoning), re-ran the tests and independently confirmed: zero cross-tenant leak (down from a
0.92 baseline measured in Deliverable 3), and — the check that actually matters — a query with no
tenant filter at all is refused by the database itself, not caught by application code.
**Verdict: APPROVE.**

## M1 begun

Work started the same day on M1 (write gate, secrets filter, single-pass fact extraction) —
`extraction/`, `write_gate/`, `secrets_filter/`. The secrets filter runs synchronously before
anything is written; extraction and the write-gate judge run in the background after the HTTP
response returns (`ADR-005`), using FastAPI's in-process `BackgroundTasks` rather than standing
up a real task queue — a deliberate "don't build infrastructure the plan never asked for" call,
flagged as something to revisit if this ever needs to survive a process crash or scale past one
instance. This work continued into the next two days (see `2026-07-15.md`, `2026-07-16.md`).
