# 2026-07-18 — M3, M4, M5, M6 closed; Deliverable 6 verification artifacts written; Deliverable 7 begun

A long single day covering four milestones plus two deliverable write-ups. Broken out by block.

## M3 — Contradiction handling (`922f555`)

Write-time contradiction detection: a new memory is compared against existing same-subject
memories (cosine similarity above `SAME_SUBJECT_THRESHOLD = 0.75`, using the embedding already
computed at write time — no re-embedding needed), exempting append-only `event`/`working` types.
The resolution rule (C8): a new memory supersedes an existing one only if it's *at least as
confident*; otherwise both stay active. The superseded row is never deleted — it's marked
`status='superseded'` with a `superseded_by` pointer and an `audit_log` entry, so history stays
auditable.

**A naming decision worth recording:** `db/schema.sql`'s status constraint uses `'superseded'`,
not `design/data_model.md`'s original `'replaced'` label — chosen instead to match the literal
wire value `api_contracts.md` specifies for the feedback response. A small inconsistency between
two design docs, resolved by picking the one that's actually load-bearing on the wire, and the
live dev database's constraint was altered directly since the test fixtures only apply
`schema.sql` once per session, not idempotently mid-development.

L4 VERIFY (live adversarial testing against real Postgres — double-supersede, an equal-confidence
tie, a lower-confidence non-supersede, cross-tenant isolation on the self-join) **approved on the
first attempt**, unlike M2. Non-blocking caveats recorded rather than fixed: the similarity
threshold is a tunable constant, and there's no `FOR UPDATE` lock on the feedback read (a
cosmetic double-audit-row risk under concurrent duplicate requests, not a correctness bug).

## M4 — Forgetting, archiving, deletion (`61443c2`)

A nightly reweighting job recomputes each memory's `weight = (importance/10) × recency_decay ×
usefulness`, with usefulness starting at a 1.0 baseline (not 0) specifically so a brand-new,
never-retrieved memory survives its first run rather than being archived immediately for having
no access history yet. A purge job hard-deletes anything already `status='deleted'`.

**Two out-of-boundary touches, both flagged before writing code:** `retrieval/search.py` needed to
start bumping `access_count`/`last_accessed_at` on returned memories — without it, the usefulness
signal would only ever move in test fixtures, never from real usage. And a new database role,
`cmis_job`, had to be created with `BYPASSRLS` (a nightly cross-tenant sweep has no tenant-registry
table to enumerate through otherwise) — narrowly scoped to exactly the grants the job needs, and
explicitly *not* the `cmis` superuser role, per a hardening note written back at M0 that flagged
`cmis` as unconditionally bypassing RLS and ruled it out for anything beyond one-time schema setup.

L4 VERIFY approved on the first attempt. One thing left open rather than solved: no scheduler is
actually wired to run the purge job on a cadence — a deployment concern outside this milestone's
scope, not a correctness gap in the job itself.

## M5 — Observability and the Deliverable-3 benchmark re-run (`129159b`)

The four health numbers (usage rate, correction rate, latency, cost) computed from real
`audit_log` and a new `request_metric` table. Cost is an estimated chars/4-token calculation
against Claude Haiku's published rate, explicitly flagged as an estimate, not a captured live
token count. `retrieve()` now times and audits every call and returns `503 memory_unavailable`
on a database connection failure instead of hanging or 500ing, bounded by a 2-second
`connect_timeout`.

**A mistake caught during the build itself, before it ever reached L4 VERIFY:** the first attempt
at `benchmark.py` re-seeded the workload with a raw `INSERT` rather than driving it through the
real write path (`write_gate.pipeline.process_turn`). That silently skipped M3's contradiction
resolution entirely — which matters because the benchmark's S2 metric (`stale_wins_rate`) is
specifically supposed to measure whether contradiction resolution works. A benchmark that bypasses
the exact mechanism it's trying to measure would have reported a misleadingly good number for the
wrong reason. Caught and fixed before submission, not after.

L4 VERIFY drove `retrieve()` against a non-routable blackhole IP (not just a refused port — a
refused port passes even without the `connect_timeout` actually doing anything) and confirmed the
503 lands in ~2.01s. Two of five non-blocking findings were fixed post-verify (a benchmark
store-size count bug; a weaker 503 test strengthened to use the same blackhole-IP method L4 VERIFY
used); three were left as documented tradeoffs.

## M6 (stretch) — Reranker and write-gate classifier prototypes (`72d261c`)

**A real spec contradiction in `PLAN.md` itself, resolved with the user before writing any code:**
the M6 entry's Outcome line described P2 as "an early trained classifier to eventually replace the
AI-judge write gate" (matching `ADR-001`), but its Success-criteria line described something else
entirely — an extraction-vs-naive-extractor comparison, matching a different, already-shipped-at-
M1 item from `research/design_backlog.md`. Rather than guessing which reading was intended, this
was asked directly — **the user chose the ADR-001 reading** (the write-gate classifier). Recorded
here because it's a real example of catching an internal inconsistency in the project's own plan
document and treating it as a decision to make with the person accountable for the submission,
not something to silently resolve either way.

**A second, more concrete gap found while scoping the classifier work:** `write_gate/judge.py`'s
docstring had claimed since M1 that "every decision is logged so it can become training data" —
checked whether that was actually true, and it wasn't. The reject branch only ever stored a reason
string, never the candidate text itself, so there was no real (text, label) dataset to train
anything on. Fixed with a new `write_gate_decision` table logging both kept and dropped decisions
with their actual text.

Two prototypes shipped, both inert by design (feature switches off regardless of outcome, per
ADR-006 and ADR-001):

- **P1 (reranker):** a genuine accuracy gain (0.00 → 1.00 top-1 accuracy across 3 scenarios) but
  at a latency cost (p95 ~178–185ms) that exceeds the ~100ms budget — **honest verdict: REJECT**,
  exactly the "either answer is fine" outcome ADR-006 was written to allow for.
- **P2 (classifier):** 1.000 held-out agreement against a 0.90 bar this milestone set for itself —
  **verdict: ADOPT** — but measured against a rule-based judge stand-in, not the real fuzzier AI
  judge, a caveat the experiment's own docstring states directly rather than burying.

L4 VERIFY approved on the first attempt, independently re-deriving expected rankings and re-running
both experiment scripts itself rather than trusting the printed verdicts. The quiz-me gate was
answered by the assistant, **at the human's explicit direction** — logged with that provenance in
`.genesis/checkpoints/M6.md` rather than presented as independent human verification. Same pattern
noted on 2026-07-15/16 about the M1 gap between AI approval and human sign-off — here the human
made an explicit, recorded choice to delegate the quiz answers rather than the gap going unnoticed.

## Deliverable 6 — writing the verification artifacts (this session)

Separately from the M0–M6 build loops above, this session did a gap analysis against the handbook
(`deliverable1 (1).pdf`) for Deliverables 5 and 6. Finding: the underlying engineering
verification work was real and rigorous (every milestone had an independent L4 VERIFY pass), but
the handbook's actual required `verification/` folder was **completely empty** — none of
`test_plan.md`, `evaluation_dataset.jsonl`, `results/`, `security_report.md`, or
`final_verification.pdf` existed anywhere in the repo, and `README.md` was missing setup
instructions and a commands list. All of the real verification evidence lived only inside
`.genesis/checkpoints/*.md`, which is Genesis's own internal record, not the standalone artifact
the handbook grades for this deliverable.

This was flagged directly rather than assumed fine, and then closed: the full test suite, mypy,
ruff, and the benchmark were all re-run fresh against a newly recreated database volume (not the
accumulated dev state), plus one additional live `psql` RLS check run independent of the pytest
suite entirely. `verification/test_plan.md`, `evaluation_dataset.jsonl`, `results/` (five files),
`security_report.md`, and `final_verification.pdf` were written from that fresh evidence.
`README.md` was expanded with an architecture summary, setup steps, and a commands table.

**Lesson for the retrospective:** doing rigorous verification work during the build is necessary
but not sufficient — if it isn't surfaced into the exact artifacts a specification requires, it
doesn't count as done against that specification, even when the substance is genuinely there.
