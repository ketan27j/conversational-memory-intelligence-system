# Baseline Protocol — Productive Failure Experiment

**Deliverable:** D3
**Author:** Ketan Juikar
**Status:** pre-registered plan (written before reading results)

This document defines the workload, the naive design, and the metrics **before** the run, so the results cannot be quietly reshaped to fit a story. The findings live in `productive_failure_report.pdf`.

---

## 1. Hypothesis

A deliberately naive memory system — store everything, rank on one similarity signal, no recency, no isolation, no admission gate, no decay, no abstention — will reproduce, as measured numbers, the failures that D1 only asserted: pollution (F2), stale-preference wins (F3), cross-tenant leakage (F5), PII retention (F6), and failure to abstain on cold-start. I expect these to **worsen as the store grows**.

## 2. Baseline design (what is deliberately simple, and why it is still fair)

The store (`naive_baseline/memory.py`) keeps every memory and ranks candidates by **TF-IDF cosine similarity** over raw text. It is "naive" by omission, and each omission maps to a capability D1 said the real system needs:

| Omitted | Capability removed (D1) | Failure it should expose |
|---------|-------------------------|--------------------------|
| Admission gate / dedup / PII filter | C3, C11 | F2 pollution, F6 PII |
| Recency / importance / frequency in ranking | C6 | F3 stale preference |
| Per-user filtering on retrieval | C10 | F5 cross-tenant leak |
| Relevance threshold / abstention | (cold-start) | injects noise when nothing fits |
| Decay / consolidation | C9 | F7 unbounded growth |

"Naive" means minimal, not rigged. TF-IDF cosine is a single, legitimate similarity signal. One honest consequence is stated up front: **TF-IDF does keyword matching, so it is actually strong on exact-string facts** (a version name, a DB name). That means the dense-embedding failure F4 (exact-fact miss) is *not* expected to appear here — the keyword arm is, in effect, the very fix D2 proposed for F4. I note this rather than hide it.

## 3. Workload (fixed, seeded)

Generated by `naive_baseline/workload.py`, `seed=42`. Eight users plus one brand-new zero-memory user. A scaled filler haystack of mundane facts creates realistic pollution; on top of it sit labelled scenario memories and a fixed probe set with ground truth. The six required scenarios:

- **S1 pollution** — a needle fact among filler ("uses PostgreSQL") + retrieval probes.
- **S2 stale preference** — an early statement then a later contradicting update (answer length, tabs→spaces, USD→INR); gold = current, poison = stale.
- **S3 budget** — measure injected tokens against a 150-token context budget; check whether the relevant memory survives.
- **S4 cross-tenant** — two users with near-identical wording, different facts; probe issued as one user.
- **S5 PII** — an API key and a card number pasted mid-conversation.
- **S6 cold-start** — probes with no relevant memory, plus a zero-memory new user.

The store is rebuilt at four sizes (≈63, 207, 783, 1935 memories) by scaling the filler, so every metric can be read as a function of store size.

## 4. Metrics

- **Retrieval quality:** precision@5, recall@5, hit@5, MRR (on answerable QA probes).
- **F3:** stale-wins rate (stale memory outranks current in the full ranking).
- **F5:** cross-tenant leak rate (top-5 contains another user's memory).
- **F6:** secrets stored (admission), PII exposure rate (secret retrievable in top-5).
- **Budget:** mean injected tokens for top-5; relevant-dropped-under-budget rate at 150 tokens.
- **Cold-start:** abstention-failure rate; mean top-1 similarity, answerable vs unanswerable (evidence of guessing).
- **Storage:** memory count, raw text KB, TF-IDF non-zero entries.
- **Latency:** p50 / p95 per-query retrieval (ms), index pre-built.

Token counts are approximated (~1.3 tokens/word); this shifts absolute token numbers slightly but not any conclusion.

## 5. Environment & reproducibility

Python 3.12, scikit-learn 1.8, numpy 2.4, pandas 3.0. Fixed seed (42), deterministic TF-IDF, fixed probe set. Re-running `python benchmark.py` reproduces `baseline_results.csv` and `error_examples.jsonl` exactly.

## 6. What would make the hypothesis wrong

If the naive store retrieved the correct memory at high precision, kept users isolated, suppressed stale preferences, and abstained on cold-start, the hypothesis fails and D1's failure analysis would need revising. That outcome is exactly what this protocol is built to allow.
