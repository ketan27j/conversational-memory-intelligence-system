## Micro-plan (produced before iter 1)

Freeze boundary per PLAN.md is `implementation/forgetting/**` + `implementation/jobs/**`.

Required touches outside that boundary, both flagged above in G0 and both necessary for the milestone's own success criteria to be real rather than fixture-only:
- `implementation/retrieval/search.py` — after `selected` (the memories actually returned to a caller) is computed, one `UPDATE memory SET access_count = access_count + 1, last_accessed_at = now() WHERE id = ANY(%s)` in the same cursor/transaction as the search itself. Runs *after* the ranking pass reads the pre-bump `access_count`, so a query never inflates its own ranking mid-request.
- `implementation/db/schema.sql` — add the `cmis_job` role (`BYPASSRLS`, `SELECT/UPDATE/DELETE` on `memory`, `INSERT` on `audit_log`, no other grants) and a matching `job_connection()` helper in `implementation/db/connection.py` (parallel to the existing `tenant_connection`/`unscoped_connection`, reading `CMIS_JOB_DATABASE_URL`). The dev Postgres volume was just recreated fresh for this session (`docker compose up -d` on an empty volume), so `schema.sql` alone covers it this time — no live `ALTER ROLE` needed, unlike M3's already-running-volume case.
- `implementation/api/main.py` — `DELETE /v1/memories/{id}` per `api_contracts.md`. Edge cases the contract doesn't literally document, decided here (same practice as M1–M3): unknown/cross-tenant id → 404 (RLS makes the two indistinguishable, no leak); already-`deleted` → 200 idempotent, no re-audit; any other non-active status (`archived`/`superseded`) is still a valid delete target (the "right to be forgotten" applies regardless of a memory's current lifecycle stage) and proceeds to mark it deleted.

Files to add:
- `implementation/forgetting/reweight.py` — `run_reweight_job(cur, now=None) -> ReweightResult`. Recomputes `weight` **fresh each run** (not an incrementally-decayed running total) from current `importance`, staleness since `COALESCE(last_accessed_at, created_at)`, and `access_count` — chosen over decaying a stored running value so the result depends only on present state, never on how many times the job has previously fired (simpler to reason about and to test deterministically). Formula: `weight = (importance / 10) * recency_decay(reference_ts) * usefulness(access_count)`, where `usefulness = 1.0 + access_count / (access_count + 5)` — a **1.0 baseline, not 0**, so a brand-new, never-yet-retrieved memory is not archived on the very first job run after it's written; access history is a bonus multiplier on top of "assumed useful," not a hard requirement. Deliberately separate constants from `retrieval/search.py`'s C6 ranking formula (different reference timestamp — staleness-since-last-useful vs. freshness-since-created — and touching M2's already-verified ranking code for an unrelated purpose is an avoidable risk). Anything landing below `ARCHIVE_FLOOR` gets `status='archived'` + an `audit_log` row; survivors get their recomputed `weight` persisted either way.
- `implementation/forgetting/purge.py` — `run_purge_job(cur) -> PurgeResult`. Hard-deletes every row already `status='deleted'` (set by the new DELETE endpoint), after writing an `audit_log` row for each. `memory_entity` has `ON DELETE CASCADE` on `memory_id` since M0, so the entity/"search index" side is cleaned up by the database itself — no separate index-purge code needed, consistent with ADR-003's one-Postgres-database design.
- `implementation/jobs/nightly_reweight.py`, `implementation/jobs/purge_deleted.py` — thin `job_connection()`-backed entrypoints (`if __name__ == "__main__":`) an external cron would invoke. No scheduler library exists in this codebase and none is added — out of scope per PLAN.md's file boundary; wiring an actual cron/systemd-timer is a deployment concern, not a code concern this milestone owns.
- `implementation/tests/test_forgetting.py` — demo command.

Demo command: `pytest implementation/tests/test_forgetting.py -v`

Success criteria being tested against (PLAN.md): store stops growing unboundedly under constant writes (reweight+archive, tested via a stale/unimportant/never-accessed memory crossing the floor); frequently-used memories are not archived by mistake (tested via a heavily-accessed memory surviving despite an old `created_at`); deletion gone-from-reads-instantly + gone-from-search-within-60s (INV-4 hard gate, tested via DELETE endpoint + `run_purge_job` without relying on a real 60-second sleep, same "test the job function directly" precedent as M1–M3 used for their own background-job-shaped work).

- decisions / deviations made this iter:
  - **New `cmis_job` role, `BYPASSRLS`, separate from both `cmis_app` and the `cmis` superuser.** Flagged in G0: there is no tenant registry table, so a cross-tenant nightly sweep has no way to enumerate tenants under RLS; M0's own hardening note rules out reusing the `cmis` superuser for anything beyond one-time schema setup. `cmis_job` gets only `SELECT/UPDATE/DELETE` on `memory` + `INSERT` on `audit_log` (+ sequence USAGE) — never exposed on any HTTP endpoint, only ever used by `implementation/jobs/**` entrypoints.
  - **`weight` is recomputed fresh each job run, not decayed incrementally.** `run_reweight_job` computes `weight = (importance/10) * recency_decay(reference_ts) * usefulness(access_count)` from the memory's *current* state every time, rather than multiplying a stored running value — the result never depends on how many times the job has previously fired, which is simpler to reason about and to test deterministically (no compounding-decay-of-decay risk).
  - **`usefulness(access_count) = 1.0 + access_count/(access_count+5)`, a 1.0 baseline, not 0.** A literal `access_count/(access_count+5)` formula (which is what `retrieval/search.py`'s existing C6 frequency signal uses) would give a brand-new, never-yet-retrieved memory a usefulness of exactly 0, and therefore a weight of 0 — meaning every new memory would be archived on the very first nightly run after it's written, regardless of importance. The 1.0 baseline treats "assumed useful until proven otherwise" as the default, with real usage as a bonus on top — this is a deliberate reinterpretation of C9's one-line formula, justified by C9's own surrounding prose ("so useful memories stay alive, and stale ones sink on their own") which implies gradual fading, not instant archival of anything unread.
  - **Deliberately did not reuse `retrieval/search.py`'s private `_recency_score`/`_frequency_score`.** C6 (ranking) decays from `created_at` ("how fresh is this") and C9 (forgetting) decays from `COALESCE(last_accessed_at, created_at)` ("has anyone needed this lately") — different reference timestamps and different purposes even though the curve shape is similar. Keeping them separate, with `forgetting/reweight.py`'s own `STALENESS_HALF_LIFE_DAYS`/`ARCHIVE_FLOOR` constants, avoids any risk of an accidental behavior change to M2's already-verified ranking code.
  - **`retrieval/search.py` now bumps `access_count`/`last_accessed_at`** for the memories actually returned (`selected`, post-ranking, post-relevance-bar, post-budget-selection), in the same cursor/transaction as the search itself, after the ranking pass has already read the pre-bump `access_count` — flagged in G0 as required, since without it C9's usefulness signal would only ever move in hand-seeded test fixtures, never from real usage.
  - **`DELETE /v1/memories/{id}` treats any non-`deleted` status (including `archived`/`superseded`) as a valid delete target**, not just `active` — the "right to be forgotten" applies regardless of a memory's current lifecycle stage. Only an already-`deleted` memory short-circuits to an idempotent 200 with no re-audit.
  - **`memory_entity`'s existing `ON DELETE CASCADE` (since M0) does the "search index" half of C11's purge requirement for free** — `forgetting/purge.py` only needs to `DELETE FROM memory`, no separate entity-table cleanup code, consistent with ADR-003's one-Postgres-database design.

### Quiz-me gate (human must answer before milestone is marked done)

1. **Design decision:** `forgetting/reweight.py` deliberately keeps its own `STALENESS_HALF_LIFE_DAYS` and a recency term measured from `last_accessed_at` (falling back to `created_at`), rather than reusing M2's already-verified C6 ranking recency formula which measures from `created_at`. Why was duplicating the decay curve with a different reference timestamp chosen over sharing one formula between ranking and forgetting?
   - Answer: The two signals answer different questions even though the curve shape (exponential half-life decay) looks the same. Ranking's C6 recency asks "how fresh is this content?", measured from `created_at` — a memory written a year ago is "old" for ranking purposes regardless of how often it's been used since. Forgetting's C9 recency asks "has anyone actually needed this lately?", measured from `COALESCE(last_accessed_at, created_at)` — a memory written a year ago but retrieved yesterday should look fresh to the forgetting job, not stale. Reusing one shared formula/timestamp for both would make one of the two questions wrong. The second reason is risk management: `retrieval/search.py`'s ranking formula was already built and independently verified in M2. Reaching into that file's private `_recency_score`/`_frequency_score` to repurpose them for an unrelated new job risks an accidental behavior change to already-approved, already-shipped code for no real benefit — a small standalone formula in `forgetting/reweight.py` with its own named constants is cheap and keeps the two milestones' verified behavior isolated from each other.
2. **Edge case:** What happens when the nightly reweight job evaluates an importance-10 memory that was created 400 days ago and has never once been retrieved (`last_accessed_at IS NULL`)? Contrast that outcome with a brand-new importance-1 memory created moments ago — why does the low-importance new one survive while the high-importance old one is archived, and is that the intended forgetting semantics?
   - Answer: The importance-10, 400-day-unused memory gets **archived**. Its reference timestamp falls back to `created_at` (400 days ago) since `last_accessed_at` is NULL, so the staleness decay has had many half-lives to erode it — `weight = (10/10) * recency_decay(400 days) * 1.0` lands far below `ARCHIVE_FLOOR` even with the maximum importance multiplier, because importance can only scale the weight, it can't stop the decay. The brand-new importance-1 memory **survives**: its reference timestamp is "now," so `recency_decay` is at its maximum (≈1.0), giving `weight = (1/10) * 1.0 * 1.0 = 0.1`, comfortably above the floor. This is the intended semantics, not a bug: importance alone was deliberately not made a permanent shield, because that would let old, genuinely-unused-but-once-rated-important memories accumulate forever, defeating the whole point of C9 ("anything that accumulates with no forgetting policy grows without limit"). Newness gets a temporary grace period so nothing is judged before it's had a chance to be used; importance only ever modulates how weight is computed, it never overrides the passage of time.
3. **Change impact:** The entity-search branch joins `memory_entity` to `memory` with `WHERE m.status = 'active'`, and the purge job relies on `ON DELETE CASCADE` rather than deleting `memory_entity` rows itself. What does that `m.status='active'` join predicate guarantee during the window after a DELETE but before the purge job runs, and what part of INV-4 would break if the entity branch queried `memory_entity` without gating on the parent memory's status?
   - Answer: Between the instant a memory is marked `status='deleted'` and the moment the purge job physically removes it (up to the 60-second INV-4 window), the row — and its `memory_entity` rows — are still physically present in the database; only the status flag has changed. The `m.status='active'` predicate on the join is what makes the entity-search branch treat that row as if it doesn't exist during that whole window, by checking the *parent* memory's current status at query time rather than assuming `memory_entity` only ever contains rows for live memories. If that predicate were removed, entity-based search would happily return a memory that's been marked deleted but not yet purged, purely because its old entity tags are still sitting in `memory_entity` — a real violation of INV-4's "reads never return a deleted memory even if the index lags," since it would mean the deleted memory *is* still reachable through one specific read path (entity search) for up to 60 seconds, while `GET /v1/memories` and the vector/keyword search branches (which already filter on `memory.status` directly) would correctly hide it. In other words: instant invisibility for entity search would start depending on how fast the purge job runs, exactly the dependency INV-4 is designed to rule out.
