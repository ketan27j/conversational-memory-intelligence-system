### Quiz-me gate (human must answer before milestone is marked done)

1. **Design decision:** The RLS policy reads tenant from a `current_setting('app.tenant_id')` session variable set via `set_config(..., false)`. Why this per-session GUC approach over giving each tenant its own Postgres role, or just binding `tenant_id` as a parameter in every `WHERE` clause?
   - Answer: A role per tenant doesn't scale operationally (provisioning/managing a Postgres role per user) and breaks connection pooling — each role needs its own connection(s), since a connection authenticated as tenant A's role can't be reused for tenant B's request. At real scale (thousands+ of end users), that means thousands of roles and no shared pool, and it doesn't actually buy more safety than the session-variable + RLS approach — the database still enforces isolation either way. The session variable lets one shared, pooled `cmis_app` connection serve every tenant safely, stamped per-request.
2. **Edge case:** A connection is opened with no `app.tenant_id` set (`unscoped_connection()`), then runs `SELECT * FROM memory`. Walk through exactly why the policy returns zero rows rather than erroring — and what would happen instead if `current_setting` were called with `missing_ok=false`.
   - Answer: `current_setting('app.tenant_id', true)` returns NULL when unset (the `true` = missing_ok). The policy becomes `tenant_id = NULL`, which SQL always evaluates to NULL/unknown — never true — for every row, so the RLS USING clause excludes everything: zero rows, fail-closed, no error. `unscoped_connection()` exists only as a test tool to simulate "the app forgot to stamp the tenant" and prove the database still refuses by default (ADR-004's point: safety can't depend on the app layer always doing the right thing). If `missing_ok=false` were used instead, Postgres would raise `unrecognized configuration parameter "app.tenant_id"` and the query would error instead of silently returning nothing.
3. **Change impact:** What does `ALTER TABLE memory FORCE ROW LEVEL SECURITY` add on top of `ENABLE ROW LEVEL SECURITY`, given that `cmis_app` is neither a superuser nor the table owner — i.e., in what deployment scenario does `FORCE` actually change the outcome?
   - Answer: `FORCE` only affects the *owner-bypass* rule — it makes RLS apply to the table owner too, which `ENABLE` alone does not. It has no effect on superuser-bypass, which is unconditional and can't be overridden by `FORCE`. Checked `pg_roles` directly: `cmis` (the table owner) is a Postgres superuser (`rolsuper=t`) — that's how the official Postgres/pgvector Docker image sets up `POSTGRES_USER` — while `cmis_app` is neither owner nor superuser. So a future script connecting as `cmis` would bypass RLS regardless of `FORCE`, because it bypasses as a *superuser*, not as the *owner*. `FORCE` would only matter in a scenario where a *non-superuser* owner role exists — e.g. a dedicated non-superuser `cmis_migrations` role that owns the tables, or if `cmis_app` were ever granted ownership directly. Hardening gap identified: nothing should ever connect as `cmis` except one-time schema setup, since that role bypasses everything unconditionally.
