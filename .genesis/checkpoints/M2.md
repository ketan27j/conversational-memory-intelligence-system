## L4 VERIFY (attempt 1) · 2026-07-16

- checker: claude-opus-4-8 (fresh context — goal + success criteria + invariants + artifact only, no build trail; explicitly instructed not to read this checkpoint file)
- commands re-run independently: pytest tests/test_retrieval.py 10/10, full suite 34/34, mypy clean (23 files), ruff clean
- verdict: **REJECT** — mechanics (parameterized queries, no SQL injection surface, tenant isolation on all three query paths including the entity JOIN, score-math bounds, empty/punctuation/long-query handling) all confirmed sound. But found a genuine, reproducible correctness bug in the relevance-bar logic itself: `search.py`'s abstention check gated on `scored[0].semantic` — the top item **by final blended score** — not on relevance per candidate. Since final score is boosted by recency/importance/frequency, an unrelated-but-recent-and-important memory can out-rank a genuinely relevant answer, then fail the bar *for the whole query*, causing a false abstention on an answerable question. Verifier reproduced concretely: seeded an unrelated "coffee machine" memory (importance=10, fresh, access_count=1000) alongside the real answer "Runs postgresql 16.4 in production" (importance=1, old, unused) — query "what version of postgresql does the user run" abstained, even though the answer alone (no distractor) scored 0.7 semantic, far above the 0.20 floor. Directly fails success criterion 2 (exact-fact questions answered correctly) and corrupts INV-6's actual intent (abstain only when nothing relevant *exists*, not when something relevant is present but outranked by noise).
- secondary (non-blocking) finding: when every relevant candidate individually exceeds `token_budget`, the function returned `abstained: False, memories: []` — a contract wart callers could crash on (`if not abstained: memories[0]`).
- **This is L2 DEBUG territory per LOOPS.md (REJECT/UNCERTAIN → spawn L2 DEBUG, retry once).** Root cause: the abstention check was applied to `scored[0]` (post-sort, post-blend) instead of filtering the whole candidate set on the semantic signal *before* ranking/selecting.
- fix applied (`retrieval/search.py`): replaced the single top-item check with `relevant = [m for m in scored if m.semantic >= RELEVANCE_FLOOR]`, abstain iff `relevant` is empty; token-budget selection now runs over `relevant` instead of all of `scored`. Also folded the secondary issue into the same fix: if budget selection empties out despite relevant candidates existing, that now also returns `abstained=True` (empty-list-implies-abstained is now a real invariant of the return value, not just the "nothing relevant" case).
- regression test added: `test_relevant_answer_survives_an_unrelated_recent_important_distractor` — exact scenario the verifier reproduced (high-importance/fresh/high-access-count distractor + low-importance/old/unused correct answer), asserts the answer is still returned.
- full suite re-run after fix: 35/35 passed (up from 34), mypy clean, ruff clean

### Quiz-me gate (human must answer before milestone is marked done)

1. **Design decision:** The relevance floor (`RELEVANCE_FLOOR`) is compared against each candidate's `semantic` score (vector+keyword+entity only), applied per-candidate *before* selection — rather than against the final blended `score` of the top-ranked result. Why was this chosen over the simpler "check if the #1 result by final score clears the bar" approach?
   - Answer: Because the final `score` also folds in recency, frequency, and importance — a memory can rank #1 overall for reasons that have nothing to do with actually answering the question (e.g. it's recent and important, but irrelevant). Gating on that #1 item would let a high-recency/high-importance distractor outrank the real answer, fail the relevance bar for the *whole query*, and cause a false abstention even though a genuinely relevant memory existed lower in the ranking. Checking `semantic` per-candidate before selection means the real answer is found and kept regardless of what out-ranks it, while anything that's truly unrelated still gets filtered out no matter how recent/important/frequent it is. This is exactly the bug the L4 verifier reproduced (coffee-machine distractor vs. the real PostgreSQL-version answer) and the fix that made attempt 2 pass.
2. **Edge case:** A query matches two genuinely-relevant memories, but every candidate's content individually exceeds `token_budget`. What does `hybrid_search` return for `abstained` and `memories`, and why is that specific combination chosen over returning `abstained=False` with an empty list?
   - Answer: It returns `abstained=True, memories=[]`. `abstained=False` signals "here's an answer," so a caller would reasonably do `if not abstained: memories[0]` and crash with an IndexError on the empty list. From the caller's point of view, "found something relevant but couldn't fit it in budget" and "found nothing relevant" both mean the same thing — no answer is being delivered — so both cases collapse to the same `abstained=True, memories=[]` contract rather than leaving a footgun case where `abstained=False` doesn't actually mean an answer is present.
3. **Change impact:** What does the `max_keyword_rank` normalization at `search.py` (in the keyword-signal block) do, and what would break in the C7 semantic blend if you removed the normalization and used the raw `ts_rank_cd` value directly as `keyword_rank`?
   - Answer: It divides every candidate's raw `ts_rank_cd` value by the highest `ts_rank_cd` seen for that query, rescaling keyword scores into a consistent 0–1 range (best match = 1.0) to match `similarity` and `entity_hit`, which are already 0–1. `ts_rank_cd` has no fixed scale — its raw magnitude varies by query/content and isn't comparable across queries. Without normalization, the `0.3` keyword weight in `semantic = 0.5·similarity + 0.3·keyword_rank + 0.2·entity_hit` would stop meaning what it's supposed to: a query where raw `ts_rank_cd` values happen to be small would make keyword matching barely move the blend at all, while a query where they're large would let keyword matching swamp the vector and entity signals entirely. Normalizing first is what keeps the three signals commensurable enough for the fixed weights to behave consistently query to query.
