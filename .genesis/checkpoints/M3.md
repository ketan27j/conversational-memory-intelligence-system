### Quiz-me gate (human must answer before milestone is marked done)

1. **Design decision:** `find_same_subject()` finds candidates by self-joining `memory` on the new row's own already-populated `embedding` column, rather than calling the embedder again on the new memory's content. Why does this work, and what would it cost (in more than one sense) to instead re-embed the content directly inside the contradiction check?
   - Answer: It works because the embedding is a fingerprint of the memory's *content*, not of *why* it's being compared — the same vector that `retrieval/indexer.py::index_memory` already computed and stored is equally valid for "does this match something in the index" (M2) and "does this match an existing same-subject memory" (M3). `find_same_subject()` runs after indexing, in the same transaction, so that vector already sits in the `embedding` column ready for a self-join. Re-embedding inside the contradiction check would cost: (1) money/latency — a second billed API call to the embedding provider for content that was just embedded seconds earlier in the same request, pure waste; (2) correctness risk — if the embedding model/version drifted between the two calls (a redeploy mid-request, provider-side model update), the new vector could differ slightly from the one already indexed, making the "same subject" comparison inconsistent with what search itself uses to find that memory later.
2. **Edge case:** Two existing active memories are both within `SAME_SUBJECT_THRESHOLD` of a new, high-confidence memory being written. What happens to each of them, and what does the `memory` table look like afterward for all three rows (`status`, `superseded_by`)?
   - Answer: Both existing memories are evaluated independently against the new one in `resolve_contradictions()`. Since the new memory's confidence is high (`>=` both candidates' confidence), both get superseded: each has `status` set to `'superseded'` and `superseded_by` set to the new memory's id, and an `audit_log` row (`action='replaced'`) is written for each. The new memory itself keeps `status='active'` and `superseded_by=NULL`. Nothing is deleted — both old rows remain in the table, just marked replaced, so their history stays queryable/auditable. This is exactly what L4 VERIFY attempt 1 independently reproduced ("one new memory contradicting two existing memories at once — both correctly superseded, both `superseded_by` point at the new memory, new stays active").
3. **Change impact:** Suppose `resolver.py`'s comparison were changed from `if new_confidence < candidate.confidence: continue` to `if new_confidence <= candidate.confidence: continue`. Concretely, what changes about the equal-confidence case, and which documented behavior (name the source) would this now contradict?
   - Answer: Currently, on an exact tie (`new_confidence == candidate.confidence`), the `<` check is false, so the loop does *not* `continue` — the new memory supersedes the old one. Switching to `<=` would make the tie case also `continue`, meaning a tied new memory would skip superseding and both memories would stay active side by side instead. This would contradict the L4 VERIFY attempt 1 finding recorded above (line 53 of this checkpoint): "equal-confidence tie (0.6 == 0.6) — new memory wins ... satisfies INV-5." INV-5 (`.genesis` invariant: newer/more-confident wins when a new memory contradicts an existing one) is defined as "newer *or* at least as confident wins," so a tie should resolve in the new memory's favor — the `<=` version would silently break that already-verified, already-approved behavior.
