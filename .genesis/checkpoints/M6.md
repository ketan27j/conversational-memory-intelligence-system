## Micro-plan (produced before iter 1)

Freeze boundary per PLAN.md is `implementation/prototypes/**`. Two required out-of-boundary touches, flagged here before writing code (same precedent M1-M5 used):

- `implementation/db/schema.sql` — new `write_gate_decision` table (`tenant_id`, `candidate_text`, `kept BOOLEAN`, `importance`, `confidence`, `reason`, `created_at`) + RLS policy. This is the actual fix for the G0 gap: a dedicated labelled-example table (not overloading `audit_log`'s free-text `detail`, same "purpose-built table over overloading an existing one" precedent M5 used for `request_metric`) that captures candidate text for **both** kept and dropped decisions, making ADR-001's "log every keep-or-drop decision as a labelled example" true for the first time.
- `implementation/write_gate/pipeline.py` — insert one `write_gate_decision` row per judge call (both branches), immediately after `judge.judge(candidate)` returns. Without this, P2 has no real (text, label) pairs to train on — same "necessary alignment, not scope creep" reasoning M2/M4 used for their own indexing/access-count touches.
- `implementation/api/main.py` — `retrieve()`'s existing `rerank` field actually does something now: when `True`, call the reranker prototype on `result.memories` before returning. Still **off by default** (`rerank: bool = False` already the default since M2) — ADR-006 requires the switch to ship off regardless of what the experiment concludes, so this touch doesn't change default production behavior, only makes the already-accepted-but-inert parameter real.

Files to add:
- `implementation/prototypes/__init__.py`.
- `implementation/prototypes/reranker.py` — `Reranker` Protocol (`rerank(query, candidates) -> list[ScoredCandidate]`) + `AnthropicReranker` (production, one LLM call per candidate scoring relevance 0-1) + a deterministic `FakeReranker` for tests (C14 — same swappable-interface-plus-fake pattern as every other LLM-backed component in this codebase), the fake modeling a realistic LLM-round-trip latency via `time.sleep` rather than being suspiciously instant, so the latency experiment is honest.
- `implementation/prototypes/classifier.py` — `NaiveBayesWriteGateClassifier`: pure-Python multinomial Naive Bayes over bag-of-words features, `.train(examples: list[tuple[str, bool]])`, `.predict(text) -> bool`. - `implementation/prototypes/rerank_experiment.py` — seeds a small store, measures accuracy proxy (MRR-style) and latency with vs without reranking, prints the ADR-006 adopt/reject verdict against the ~100ms budget.
- `implementation/prototypes/classifier_experiment.py` — seeds N labelled write-gate decisions through the real write path (a deterministic rule-based `FakeJudge` standing in for "the AI judge," C14-style — no live ANTHROPIC_API_KEY dependency for the demo), trains the classifier on a train split, measures agreement with the judge on a held-out split, compares cost (classifier ~$0 vs judge's `estimate_llm_cost_usd`), prints the ADR-001 adopt/reject verdict against a documented bar (this milestone sets ≥90% held-out agreement as "worth adopting," flagged as this milestone's own starting guess — ADR-001 gives no numeric bar).

### Quiz-me gate

**Process note:** per LOOPS.md, these three questions are meant to be posed to a human and answered independently, as a check that a human (not just the AI reviewer) understands and endorses the reasoning. The human present for this milestone (ketan) explicitly directed the assistant to explain and answer them instead of answering independently — logged here verbatim as asked, with that provenance noted rather than presented as unprompted independent human verification.

1. **Design decision:** Why did the first attempt at `rerank_experiment.py`'s scenarios produce a meaningless result, and what had to be true of the distractor text to fix it?
   - Answer (assistant, at human's direction): The first attempt used distractor text sharing only 2 of the 5 query terms (e.g. "database" and "version" but not "user", "run", "production"). Postgres's `plainto_tsquery` ANDs every non-stopword term together — a row only enters the keyword-scored candidate pool at all if its `content_tsv` matches *all* of them. Missing 3 of 5 required terms meant the distractor's keyword score stayed at zero; it never competed on that signal at all. Combined with the true answer's perfect (cosine = 1.0) vector match, the true answer won regardless of reranking — baseline accuracy was already 1.00, so there was nothing for reranking to improve. This was verified directly against Postgres's actual `to_tsvector`/`plainto_tsquery`/`ts_rank_cd` output, not assumed from reading the SQL. The fix: rewrite distractor text to contain literally every stemmed term the tsquery requires, so it enters the keyword pool with a normalized `keyword_rank = 1.0`, and combined with a high importance value (8 vs 3), its blended score can genuinely beat the true answer's vector-only score under C6's weights (0.4 semantic / 0.2 recency / 0.2 frequency / 0.2 importance) — producing a real miranking (baseline accuracy 0.00) for reranking to fix.

2. **Edge case:** What does `NaiveBayesWriteGateClassifier.predict()` return when called on text with zero training data (`train([])` was just called), and why?
   - Answer (assistant, at human's direction): It deterministically returns `True`. With no training data, `total_docs == 0`, so `predict()`'s prior calculation falls through to a fixed `1e-9` for both labels (the `total_docs and doc_count` guard is false either way). Every word in the input is unseen, so each class's word-likelihood term reduces to `log((0+1)/(0+vocab_size))` — identical for both classes since `word_totals` is 0 for both and `vocab_size` is the same shared value. The two labels end up with exactly equal log-probability. The tie isn't broken randomly — it's an artifact of iteration order and a strict `>` comparison: the loop checks `True` first, sets it as best, then checks `False`, which doesn't beat (only ties) the current best, so `True` never gets overwritten. Not a crash, not arbitrary — a deterministic tie-break baked into the loop structure.

3. **Change impact:** What actually changed about `retrieve()`'s error-handling scope now that reranking runs after the `tenant_connection` block closes, and why couldn't the code just do `result.memories = reranked`?
   - Answer (assistant, at human's direction): Before reranking existed, everything in `retrieve()` ran inside one `try: with tenant_connection(...): ... except psycopg.OperationalError: raise 503` block. Placing reranking outside that block means a reranker failure (a bad Anthropic API response, a timeout) is **not** caught by `except psycopg.OperationalError` and does **not** get mapped to `503 memory_unavailable` — it propagates as a genuine 500. That's the correct behavior: C13/INV-7 specifically promises graceful degradation for *memory store* failures, not reranker failures, and conflating the two would misreport an LLM-side error as a database outage. It also means the DB connection is held open only for the search + audit + metric write, then released before any (potentially multi-round-trip) LLM calls — no connection sits idle across reranking latency. As for the direct-assignment question: `RetrieveResult` is a frozen dataclass (`@dataclass(frozen=True)`), so `result.memories = reranked` would raise `dataclasses.FrozenInstanceError` at runtime — frozen dataclasses forbid attribute mutation by design, the same reason `reranker.py` itself uses `dataclasses.replace()` on `ScoredMemory` rather than mutating it. The endpoint instead reassigns a local `memories` variable and builds the response from that.
