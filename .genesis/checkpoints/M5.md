### Quiz-me gate

**Process note:** per LOOPS.md, these three questions are meant to be posed to a human and answered independently, as a check that a human (not just the AI reviewer) understands and endorses the reasoning. The human present for this milestone (ketan) explicitly directed the assistant to answer them instead of answering independently — logged here verbatim as asked, with that provenance noted rather than presented as unprompted independent human verification.

1. **Design decision:** `benchmark.py` seeds memories by calling `write_gate.pipeline.process_turn` (with a fixed single-candidate extractor + always-keep judge) instead of a raw `INSERT INTO memory`. Why does that choice matter for the S2 (stale-preference) comparison against the baseline?
   - Answer (assistant, at human's direction): `process_turn` runs the *entire* write path, not just the insert — extraction (stubbed here), the judge (stubbed to always keep), the insert itself, `retrieval/indexer.py::index_memory` (embeds + populates `memory_entity`), and — critically — `contradiction/detector.py::find_same_subject` + `contradiction/resolver.py::resolve_contradictions` (M3, C8), which is what marks a superseded "stale" memory `status='superseded'` so `hybrid_search`'s `WHERE status='active'` filter excludes it from results entirely. A raw `INSERT` skips `find_same_subject`/`resolve_contradictions` completely, since neither lives in the INSERT path — the stale memory would stay `active` forever and remain eligible to outrank the current one. Since S2's `stale_wins_rate` is specifically measuring whether M3's contradiction resolution keeps a stale preference from beating the current one, bypassing the write path would make the benchmark measure nothing about M3 at all — it would just be testing raw retrieval ranking on two co-existing, uncontested memories. This was caught empirically, not by inspection: an early bypass-the-write-path version of the script produced a real-system `stale_wins_rate` of 0.667, *worse* than the naive baseline's 0.333, which directly contradicted M3's own passing test suite (`test_contradiction.py`) — that contradiction was the tell that the benchmark, not M3, was broken.

2. **Edge case:** In `retrieve()`, the audit-log insert and the `request_metric` write happen *inside* the same `tenant_connection` block as the search, right after the search succeeds. What happens to the response if the database connection fails on one of those two writes, after the search itself already returned results?
   - Answer (assistant, at human's direction): The whole block — the `hybrid_search` call, the `audit_log` INSERT, and the `record_metric` INSERT — runs inside one `tenant_connection` context and one implicit transaction. If the connection fails on either of the two writes (after `hybrid_search` has already succeeded and computed `result` in Python), `tenant_connection`'s own exception handler (`db/connection.py`) rolls back the transaction and re-raises; `retrieve()` catches that as `psycopg.OperationalError` and returns `503 memory_unavailable`, exactly the same response as if the search itself had failed. The already-computed `result` object is simply discarded — the function never reaches `return RetrieveOut(...)`. So the caller gets a 503 even though the actual retrieval succeeded; the failure is in bookkeeping (audit/metrics), not retrieval. This doesn't violate C13/INV-7 — the turn is never failed, the caller still gets a clean signal to answer without memory — but it is slightly pessimistic: a successful search result is thrown away because of an unrelated write failure a few lines later. Flagged as a known, narrow, non-blocking edge case rather than fixed, since fixing it (e.g. writing audit/metrics best-effort, outside the retrieval-critical path) is a real design change, not a one-line patch.

3. **Change impact:** `db/connection.py` gained `CONNECT_TIMEOUT_SECONDS = 2`, applied only to `tenant_connection`. What's the practical effect of that on `retrieve()`'s behavior when the database is unreachable — and why was it deliberately *not* added to `job_connection` or `unscoped_connection`?
   - Answer (assistant, at human's direction): `CONNECT_TIMEOUT_SECONDS` is passed as `connect_timeout=2` to `psycopg.connect(DSN, ...)` inside `tenant_connection` — the function every live API call goes through (ingest, retrieve, feedback, delete, list, health). Without it, a connection attempt to a down or black-holed Postgres would hang on the OS/libpq default TCP timeout, which for a non-routable address (as opposed to an actively-refused port) can be tens of seconds to minutes — L4 VERIFY's own adversarial test against a blackhole IP (`10.255.255.1`) confirmed the *unbounded* case would otherwise hang, and confirmed the bounded case returns in ~2.01s. With the timeout, `retrieve()`'s outage path degrades in ~2 seconds instead of hanging indefinitely, which is what C13/INV-7 actually requires ("the lookup times out and the assistant answers without memory" — a hang would violate that promise just as badly as an unhandled exception would). It was deliberately *not* added to `job_connection` because that connection is only ever used by the M4 nightly forgetting/purge background jobs (`implementation/jobs/**`), which run off the live request path entirely — nothing user-facing is waiting on them, so a hang there is a different failure mode (a stuck cron job) that this milestone didn't take on. It was not added to `unscoped_connection` because that connection exists purely as a test fixture (`tests/test_tenant_isolation.py`) to prove RLS refuses an unfiltered, no-tenant connection — it is never used on any real request path, so a production-tuned timeout there would be scope creep with no functional benefit and isn't part of the C13/INV-7 contract at all.
